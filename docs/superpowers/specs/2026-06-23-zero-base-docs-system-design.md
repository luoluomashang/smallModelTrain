# Zero-Base Documentation System Design

## Approval Context

The user asked to sort out the project code and design, then supplement documentation that a complete beginner can understand. After reviewing the repository structure, existing README, stage guides, scripts, modules, tests, and recent commits, the selected direction is the full documentation cleanup: a zero-base entry, code/design guide, pipeline guide, troubleshooting guide, glossary, and document index.

## Goal

Create a beginner-friendly Chinese documentation system for Small Model Train so a reader with no prior context can understand what the project does, where files live, how data flows, which commands to run first, how the code is organized, and where to look when something fails.

## Current Project Shape

Small Model Train is a Python project for a Qwen3-4B-Instruct QLoRA pipeline that trains and evaluates a Chinese whole-chapter novel executor. The repository already has working stage runbooks and implementation history, but the current entry points are command-heavy and assume the reader already understands terms such as SFT, QLoRA, adapter, chapter card, eval card, JSONL, smoke training, and agent review.

The current structure is broadly:

- `README.md`: top-level command list and stage links.
- `scripts/`: command-line entry points for data preparation, training checks, smoke/full training, inference, scoring, reporting, and review.
- `src/small_model_train/`: reusable implementation logic used by the scripts.
- `tests/`: pytest coverage for data, training guards, scoring, reports, quality gates, and review logic.
- `configs/`: training and inference configuration files.
- `data_raw/`, `data_clean/`, `data_cards/`, `data_sft/`: staged data directories.
- `outputs/`, `reports/`, `logs/`: generated model outputs, human-readable reports, and execution logs.
- `docs/`: stage guides, decision logs, summaries, and Superpowers specs/plans.

## Recommended Approach

Use a layered documentation system instead of turning the README into a very large manual.

The README remains short and useful as the front door. Detailed beginner material moves into focused Chinese docs under `docs/`. Existing stage guides stay valid and are connected through a new index and reading order.

This approach keeps each document easy to read and maintain:

- Beginners can follow a single learning path.
- Experienced users can jump to commands, reports, or troubleshooting.
- Future contributors can update one focused document instead of editing a large mixed-purpose README.

## Alternatives Considered

### Single Giant README

This would put most explanations directly into `README.md`. It is easy to discover but would become long, repetitive, and harder to maintain. It also makes the first screen noisier for experienced users.

### Minimal Index Only

This would add a navigation page over the existing documents. It is quick, but it does not satisfy the zero-base requirement because many core concepts are still unexplained.

## Documentation Architecture

### `README.md`

Purpose: project front door.

Content:

- One-paragraph explanation of the project in plain Chinese.
- "If you are new, start here" links.
- Minimal quick-start sequence.
- Links to stage guides and generated reports.
- A short note that real GPU training needs the local model and training environment.

The README should not duplicate every detailed command from the deeper docs. It should route readers to the right guide.

### `docs/index.zh.md`

Purpose: document map and reading order.

Content:

- Beginner reading path.
- Operator reading path for people who only want to run the pipeline.
- Developer reading path for people who want to modify code.
- Existing stage guide index with one-line descriptions.
- Generated artifact index: where reports, logs, outputs, and datasets are expected.

### `docs/zero-start.zh.md`

Purpose: complete beginner manual.

Content:

- What this project is and what it is not.
- Required background explained in plain language.
- How to open PowerShell in the repository.
- How to install/use Python dependencies for the lightweight project checks.
- How to prepare raw novel text.
- The safest first commands to run.
- The difference between dry-run, data preparation, smoke training, full training, inference, scoring, and review.
- For each beginner workflow: input, command, output, success sign, common failure sign.

This doc should assume the reader has never used this repository before.

### `docs/project-map.zh.md`

Purpose: explain the repository layout.

Content:

- Directory-by-directory explanation.
- Which folders are source files and which are generated artifacts.
- Which files are safe to edit manually and which are generated by scripts.
- How JSONL files are used in this project.
- How `configs/`, `reports/`, `logs/`, and `outputs/` relate to each other.

### `docs/pipeline-flow.zh.md`

Purpose: explain the end-to-end data flow.

Content:

- Flow from raw novel text to cleaned chapters.
- Flow from chapter cards to SFT dataset.
- Flow from training config to adapter output.
- Flow from eval cards to generated outputs, metrics, quality reports, and agent review.
- Stage boundaries: Stage 1, Stage 2, Stage 3, Stage 4, Stage 4.1.
- A compact Mermaid diagram or plain text diagram that renders in Markdown.

### `docs/code-design.zh.md`

Purpose: explain code organization and design for future maintainers.

Content:

- `scripts/` as thin CLI entry points.
- `src/small_model_train/` as reusable logic.
- `tests/` as behavior protection.
- Module groups:
  - text/data preparation helpers.
  - chapter card and execution card validators.
  - SFT dataset builder.
  - Stage 2 model/env/training/inference/adapter helpers.
  - Stage 3 data readiness helpers.
  - Stage 4 quality and agent review helpers.
  - scoring and reporting helpers.
- How a new script should be structured in this repository.
- How a new test should be added.

### `docs/troubleshooting.zh.md`

Purpose: beginner-friendly failure diagnosis.

Content:

- JSONL malformed data errors.

- Empty or too-short generated output.
- Scoring/reporting mismatch.
- Agent review artifact mismatch.
- Which log/report file to inspect for each failure class.

Purpose: plain-language terminology reference.

Content:

- Execution card.
- Chapter card.

Each term should use short, practical explanations tied to this project.

## Data Flow

The beginner docs should describe the pipeline in this order:

1. Put raw novel text under `data_raw/novels/`.
2. Build `data_clean/chapters_raw.jsonl` with `scripts/ingest_raw_text.py`.
3. Build `data_clean/chapters.jsonl` with `scripts/clean_chapters.py`.
4. Split train/eval rows into `data_clean/chapters_split.jsonl` and eval cards.
5. Build `style_contract.md` and `style_profile.json`.
6. Build or repair chapter cards under `data_cards/`.
7. Build `data_sft/sft_chapter_v1.jsonl` and optional LLaMA-Factory dataset metadata.
8. Run readiness checks before training.
9. Run smoke training before any full training.
10. Check adapter artifacts.
11. Run eval inference.
12. Score generated outputs.
13. Build evaluation and quality reports.
14. Run agent review gates where applicable.
15. Use decision logs to decide whether to expand data or training scale.

## Error Handling Documentation

The docs should avoid vague advice like "fix the data" or "check the logs." Each failure entry should say:

- What the symptom looks like.
- What it usually means.
- Which command or file produced it.
- Which file to inspect next.
- What a beginner can safely try.
- When to stop and not retry blindly.

GPU training failures should direct users to the event log, stdout/stderr logs, OOM probe report, model check report, and training environment report. Data failures should direct users to the exact JSONL artifact and validation command.

## Testing And Verification

Because this is documentation work, verification focuses on consistency and linkability:

- Run `pytest` if runtime is reasonable in the local environment.
- Search for stale or broken doc references to newly added files.
- Search for placeholder words such as `TODO`, `TBD`, and "待补".
- Confirm every new document is linked from `README.md` or `docs/index.zh.md`.
- Confirm commands shown in beginner docs point to existing scripts and known files.
- Confirm no doc claims that real GPU training was rerun as part of this cleanup.

## Out Of Scope

This documentation pass will not:

- Change training, inference, scoring, or review logic.
- Rename existing data directories.
- Regenerate real training outputs.
- Run long GPU jobs.
- Rewrite all historical Superpowers specs/plans.
- Remove existing stage guides.
